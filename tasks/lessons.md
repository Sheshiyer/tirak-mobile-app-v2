# Lessons

- Preview auth and onboarding screens need an actual visual anchor plus light motion before release review. Do not rely on static icon cards for the first role-selection decision; verify the simulator screenshot for perceived energy, footer spacing, and touch target comfort.
- Keep test/demo account access out of visible production UI. Use the backend-backed test accounts from `MEMORY/WORK/20260518-160000-test-accounts/PRD.md` and simulator automation to prefill/login during release testing instead of adding demo buttons or account pickers to the app.
- Do not invent profile photos for real user/guide accounts during preview cleanup. Fallback media is acceptable only for explicitly marked test/demo accounts; real profiles with missing backend images should keep the app placeholder until the backend/user supplies a real image.
- Empty profile sections are part of onboarding, not dead space. When the guide owns the profile, missing bio, languages, services, photos, and reviews should point to the exact update surface; when viewed by travelers, missing real data should be framed honestly without edit affordances.
- Before screenshot capture, scan visible text for leaked i18n keys and overly loud fallback copy. Missing prices should read as quiet secondary metadata, not as primary branded CTA text.
- Avoid marketplace copy that can be misread as adult-service pricing. For unset guide prices, use travel-context language such as "Request guide rate" and keep it visually secondary.
- Demo data should be scoped to explicit test accounts and should cover the full preview journey: profile details, services, reviews, availability, booking, and chat. Do not let UUID-shaped companion IDs be mistaken for chat room IDs.
- Demo availability must be deterministic and selectable for explicit test guides before screenshot capture. Empty backend availability is valid for real accounts, but release-preview accounts need enough seeded journey data to reach booking.
- Demo booking submission should stay local for explicit test companions when backend review endpoints are incomplete. The preview flow needs to reach confirmation without creating fake production data or failing on expected 404s.
- Booking confirmation should focus on next action, not generic utility clutter. After confirmation, show the guide, chat/message, countdown to the booked time, and Add to Calendar; avoid call CTAs unless the backend/user has a real phone interaction model.
- Demo booking coverage must include list, create, and detail reads. If a preview booking uses a local `demo_booking_*` ID, every screen that opens that ID should resolve local detail data instead of falling through to backend 404.
- Profile edit actions must not be placeholder alerts in preview-critical flows. Camera/Gallery options need to invoke the native picker, update the visible image immediately, and persist to auth/profile cache even when the backend update endpoint is unavailable.
- After saving a picked profile photo, do not immediately refetch stale backend profile data over the selected image. The just-saved image should remain authoritative in auth state and profile cache until the backend can return the same asset.
- Profile photo screens must share one authoritative local value after an edit. If auth/menu state has the newly picked photo, profile edit and main profile screens should prefer it over stale customer-profile query data.
- When the user confirms a just-fixed flow is working in the simulator, skip redundant verification and move to the next requested capture path.
- Demo booking flows must emulate the production contract end to end. If traveler booking creation is local for preview, companion booking lists, details, accept, and decline must read/write the same local booking cache while real IDs continue using the backend endpoints.
- Currency conversion should not depend on a runtime external API for preview-critical booking flows. Carry the service's source currency through booking state, convert to traveler display currency locally, and show the original guide rate when it differs.
- Companion profile editing must follow the same persistence model as traveler profile editing: native camera/gallery actions, immediate auth-state update for the drawer, query-cache update for edit/profile screens, and no stale refetch overwriting local profile media or identity fields.
- Avoid parallel companion profile flows. Dashboard "Enhance Profile", settings profile edit, and profile completion should all route to the same canonical edit screen and share the same backend-compatible payload mapping.
- Dashboard preview metrics should be derived from API/domain data, not fixed numbers. If `/suppliers/stats` is unavailable in dev, derive counts, ratings, and weekly earnings from bookings/profile fallback data so the concept still matches production.
- Backend-first preview fallbacks must still emulate production contracts. Availability, experiences, analytics, bookings, and profile edits should try the real endpoint first, normalize realistic response shapes, then persist local preview state only when the backend endpoint is missing or unavailable.
- Successful companion setup actions should return the user to the dashboard when the task is complete. Do not leave preview users stuck on setup entry screens after a save alert.
- Booking list and detail screens must choose the displayed person by viewer role from the same booking object. Companion-facing bookings show the traveler; traveler-facing bookings show the guide, and both card and detail must use the same image/name source.
- Button labels in paired mobile action rows need explicit responsive room. Longer labels like "Add to Calendar" should get wider flex allocation or smaller type before simulator capture.
- Do not leave obsolete mock-only routes reachable after a backend-first flow exists. If supplier request screens remain in navigation, adapt real booking API data into their UI contract rather than rendering `mockBookingRequests`.
- Authenticated layouts must redirect when auth state becomes unauthenticated. A null user after logout is not a loading state; it needs a deterministic `/auth` transition or the app appears hung.
- Expected dev/backend-missing fallbacks should not log Axios errors before returning fallback data. Classify the response first, then log only real failures that the user can act on.
- Preserve explicit Apple review demo accounts while removing accidental mock-only UI. Demo data should live behind named test/demo flows or backend-first fallback clients, not in reachable release screens as static mock arrays.
- Shared profile placeholders must be neutral, not real-person stock photos. Real profile screens without a backend image should show the app avatar placeholder until the user uploads an image.
- Apple review/auth preview accounts must not be blocked by phone OTP gates. If the product does not have a complete verification UX in the release path, backend registration/login should create or promote review accounts to active state instead of showing phone-verification errors.
- Registration QA should use fresh throwaway traveler and companion accounts, prove persistence through login/refetch, then immediately hard-delete those QA accounts from the backend. Do not use or delete the Apple review demo accounts for destructive verification.
- A migration-history fix is not proof that live D1 schema matches the app contract. After repairing Wrangler bookkeeping, smoke-test every preview-critical endpoint against live D1 and use Worker tail for any 500 before starting screenshots.
- Customer and supplier profile image columns are intentionally different in live D1: customers use `customer_profiles.profile_image`, while suppliers use JSON `supplier_profiles.profile_images`. Do not reuse supplier gallery parsing against customer rows.
- Companion detail, availability, bookings, notifications, and analytics are release-gating endpoints. If any of them log Axios/backend fallbacks during simulator preview, verify the live table shape before assuming the mobile fallback is acceptable.
- Chat release surfaces should expose only backend-backed actions. Until native media upload, voice messages, emoji picker, phone calls, and video calls are implemented end to end, keep conversation lists and chat composers to text chat plus send only.
- Booking review and detail screens must resolve identity from the selected booking object, not from static mock companion arrays. A mismatched guide name/photo in summary/detail is a release-blocking trust issue even when the booking itself succeeds.
- Persisted booking state may store display labels while step UIs expect internal ids. Re-entered booking steps must normalize display labels back to ids before rendering options, otherwise the flow silently loses meeting-point context.
- Explicit Apple review chat targets must route to deterministic review rooms before touching the live chat backend. If the test companion is routed into a real room with mismatched participants, the UI may appear to send while the backend returns 500.
- Invalid-token cleanup should not emit raw API JSON on public auth/onboarding screens. Once a 401 has been classified and auth is being cleared, avoid `console.error`/`logger.error` paths that create visible red development banners.
- Demo prefill route params must not create fresh objects on every render. Memoize derived review-account data before using it in a state-writing effect, or deep links like `?demo=companion` can trigger an infinite render loop.
- Supplier chat should not keep a separate mock-only conversation implementation once the shared chat route exists. Route supplier conversation cards into the same backend/demo-aware chat surface so traveler and companion behavior stays consistent.
- Live D1 migration bookkeeping is not enough for chat health. Before calling real chat healthy, verify `chat_messages` has the exact columns the Durable Object inserts (`delivered_at`, `read_at`, `reply_to_id`) and smoke-send a real backend room, because a marked-applied migration can still leave schema drift that only appears as `500 Message failed`.
- Notification read/write failures should degrade locally before they reach the UI. If a backend notification endpoint is missing or temporarily failing during preview, mark read state optimistically and persist local read ids so the bell badge/list does not show raw Axios banners.
- Booking cards must separate booking state from payment/cash state. Do not show payment `Pending` beside a `Confirmed` badge without context; use guide-rate/cash copy and reserve booking status badges for actual approval status.
- Companion approval controls must live wherever a pending booking can be inspected. Dashboard shortcuts are not enough; booking list cards and booking detail screens need the same approve/reject actions wired through the shared booking status mutation.
- Booking verification icons should come from real profile fields only. Do not assume every booking participant is verified, and avoid stacking a profile check mark beside a confirmed-status check on compact booking cards.
- Message list cards should never expose raw account handles when a friendly display can be derived. Normalize dotted/email-style names to first-name labels and show a short real last-message preview so the card feels like a live chat channel, not an account table.
- Before editing a screenshot-referenced surface, verify the route/source file that actually renders it. A matching image asset or visual fragment is not enough context, especially when multiple Tirak chats/repos are active.
- Live list endpoints may validate query-string values before coercion. Do not send numeric/boolean pagination or range filters from mobile unless the Worker explicitly accepts string coercion; strip unsupported params or retry safely without filters.
- Creating a public GitHub repository is not the same as publishing the code. For public repo handoff, scan for tracked secrets and large build artifacts first, push only a sanitized clean history, and verify the remote default branch exists before telling the user the repo is populated.
